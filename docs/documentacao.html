Carlesso Pilates API

API REST para gestão de pacientes, profissionais, planos de pagamento, aulas e prontuário clínico do estúdio Carlesso Pilates. Construída com Spring Boot 3 e Java 21, persistência em PostgreSQL, autenticação JWT stateless e exportação de relatórios em PDF/XLSX.

Java 21 Spring Boot 3.4.5 PostgreSQL 16 JWT Flyway Docker

1 Objetivo

API REST para gerenciar pacientes e profissionais de um estúdio de pilates. Oferece:

Pacientes & Profissionais
Cadastro, consulta paginada e filtrada, atualização parcial e soft delete.
Planos & Cobranças
Planos de pagamento (mensal/trimestral/anual), cobranças automáticas e controle de vencimentos.
Aulas
Geração automática conforme frequência semanal e marcação de presença vinculada ao profissional.
Prontuário Clínico
Anamnese, avaliações fisioterapêuticas, planos de tratamento, sessões, evoluções e reavaliações.
Relatórios
Relatório de pagamento por profissional (JSON/PDF/XLSX) e relatório de NFSEs por competência (JSON/CSV/XLSX).
Dashboard
Resumo consolidado com contadores e indicadores financeiros do mês corrente.

3 Arquitetura em camadas

A aplicação segue o padrão clássico Controller → Service → Repository, com DTOs como contrato externo e exceções customizadas mapeadas para status HTTP.

Cliente Frontend Angular ou consumidores externos (curl, scripts, Postman).
▼
Security Filter JwtAuthenticationFilter valida o Bearer token a cada requisição; SecurityConfig define regras de acesso e CORS.
▼
Controller Camada REST com @RestController, validação (@Valid) e mapeamento HTTP. Trabalha apenas com DTOs.
▼
Service Regras de negócio e transações (@Transactional). Lança exceções de domínio (ResourceNotFound, Conflict, Business).
▼
Repository Spring Data JPA com queries derivadas e @Query consolidadas para evitar N+1.
▼
PostgreSQL Migrações estruturais por Flyway (db/migration) e seeds apenas em dev (db/seed).

Componentes transversais

GlobalExceptionHandler
Mapeia exceções para HTTP 400/404/409/422/429. Resposta padronizada {"erro":"..."}.
OpenApiConfig
Configura Swagger UI em /swagger-ui.html e OpenAPI JSON em /api-docs.
CobrancaScheduler
Job diário que vence pagamentos e gera cobranças futuras (cron parametrizável).
LoginAttemptService
Rate limiting in-memory por e-mail: 5 tentativas falhas / 15 minutos → 429.

5 Autenticação & Usuários

Autenticação stateless via JWT. Senhas armazenadas com BCrypt. Roles: USER e ADMIN. Tokens incluem tokenVersion e são invalidados após troca/redefinição de senha.

Fluxo

  1. POST /auth/login valida credenciais e retorna access token. Após 5 falhas em 15 min retorna 429. A criação de usuários é restrita a administradores em POST /users; não há registro público.
  2. Requisições autenticadas devem enviar Authorization: Bearer <accessToken>.
  3. Tokens emitidos antes da última troca/redefinição de senha retornam 401.
  4. Endpoints /admin/** e o CRUD de /users exigem role ADMIN → caso contrário, 403.

9 Regras de negócio

Pacientes

  • Apenas um plano ativo por vez por paciente.
  • Pacientes inativos não recebem cobranças nem têm aulas geradas.
  • Soft delete preserva histórico clínico/financeiro.
  • Unicidade parcial: email e cpf únicos quando preenchidos (índices parciais WHERE col IS NOT NULL).

Profissionais

  • Tipos de contrato: CLT, PJ, AUTONOMO.
  • percentualPagamentoAula = percentual do valor da aula que o profissional recebe.
  • Relatório de pagamento considera aulas realizadas no período e ignora aulas de pacientes inativos.
  • Consulta consolidada com JOIN + GROUP BY para evitar round-trips.
  • Cálculo: valor pagamento / qtde aulas pagamento × percentualPagamentoAula / 100.
  • Limites de proteção: até 366 dias e até 5.000 aulas por relatório.

Planos de pagamento

  • Tipos: MENSAL, TRIMESTRAL, ANUAL.
  • Quantidade de dias da semana selecionados deve corresponder exatamente à frequência (1x, 2x ou 3x).
  • Criar novo plano inativa automaticamente o anterior.

Frequência de aulas

FrequênciaVezes/semanaAulas/mês (referência)
UMA_VEZ14
DUAS_VEZES28
TRES_VEZES312

Pagamentos

  • Status: PENDENTE → PAGO ou VENCIDO.
  • Valor não pode ser menor que o do plano.
  • Não pode haver dois pagamentos para o mesmo plano no mesmo período.
  • Ao confirmar, as aulas do período são geradas automaticamente.
  • dataPagamento é opcional no corpo da confirmação (default = hoje).

Geração de aulas

  • Baseada nos dias da semana do plano e no período do pagamento.
  • Idempotente: aula do paciente já existente naquela data é ignorada.
  • Requer paciente ativo e pagamento confirmado.
  • Consultas filtram apenas aulas de pacientes ativos.
  • Ao realizar, profissionalId pode ser informado para alimentar o relatório de pagamento.

NFSE

  • Considera apenas pagamentos PAGO com dataPagamento dentro da competência MM/AAAA.
  • Pacientes inativos são ignorados.
  • DescricaoServico gerada como Aulas de Pilates - Competência MM/AAAA.
  • NotaAnteriorEmitida inferida pela existência de pagamento confirmado anterior do mesmo paciente.
  • Registros incompletos retornam 422.

Avaliações fisioterapêuticas

  • Múltiplas avaliações por paciente (histórico clínico).
  • Criar para paciente inexistente/inativo → 404.
  • Obrigatórios: dataAvaliacao, queixaFuncional, escalaDor, diagnosticoFisioterapeutico.
  • escalaDor: inteiro 0–10.
  • Atualização parcial: apenas campos não-nulos do DTO são aplicados.

Planos de tratamento

  • Múltiplos planos por paciente (histórico).
  • Obrigatórios: pacienteId, dataInicio, objetivosTratamento.
  • dataFimPrevista, se informada, não pode ser anterior a dataInicio.
  • numeroSessoesPrevistas: somente positivo quando informado.
  • DELETE é lógico (preserva histórico).

Sessões de Pilates/Fisioterapia

  • Status default: AGENDADA.
  • Transições permitidas: AGENDADA → REALIZADA e AGENDADA → CANCELADA (via PATCH).
  • PUT faz atualização parcial mas não altera status.
  • Evolução clínica estruturada deve ser registrada em /evolucoes-sessao.
  • Excluir uma sessão remove também a evolução vinculada.

Evoluções de sessão

  • No máximo uma evolução por sessão (unicidade por sessao_id).
  • Segunda evolução para a mesma sessão → 409.
  • Obrigatórios: sessaoId, dataHoraRegistro.
  • dorAntes/dorDepois: inteiros 0–10 (quando informados).

Boas práticas Spring

  • Métodos de leitura nos services usam @Transactional(readOnly = true).
  • DTOs como contrato externo — entidades JPA nunca são expostas diretamente.
  • Repositórios usam @Query consolidada quando há risco de N+1.

10 Scheduler (processos automáticos)

O CobrancaScheduler executa dois jobs diários parametrizáveis via env.

Horário (default)AçãoConfiguração
06:00 Marca como VENCIDO pagamentos PENDENTE com vencimento passado. app.cobranca.cron-vencidos · APP_COBRANCA_CRON_VENCIDOS
07:00 Gera cobranças futuras para planos ativos, a partir de 7 dias antes do fim do período. app.cobranca.cron-cobrancas-futuras · APP_COBRANCA_CRON_COBRANCAS_FUTURAS

O vencimento de cobranças geradas é definido por app.cobranca.vencimento-dias (env APP_COBRANCA_VENCIMENTO_DIAS, default 10), somado ao início do período.

13 Banco & migrações (Flyway)

14 Importação de pacientes (seufisio)

O script scripts/import_seufisio.py popula o ambiente com a base real de clientes vinda do seufisio.com.br.

O que o script faz

  1. Consulta GET /api/cliente?per_page=500 no seufisio com Bearer token capturado do navegador.
  2. Para cada cliente, busca o detalhe e mapeia para o contrato de POST /pacientes.
  3. Faz login na API local, carrega CPFs já cadastrados (idempotente) e cria apenas os novos em lote.
  4. Pacientes com situacao != 2 são marcados como inativos via PATCH /pacientes/{id}/inativar.

Configuração

# scripts/.env (NUNCA commitar)
SEUFISIO_TOKEN="eyJ0eXAi..."
SEUFISIO_CLINICA_ID="..."
SEUFISIO_VERSION_APP="..."
LOCAL_API_URL="http://localhost:8080"
LOCAL_EMAIL="admin@carlessopilates.com"
LOCAL_PASSWORD="senha1234"

Execução

set -a; source scripts/.env; set +a

# Dry-run (recomendado antes da carga real)
python3 scripts/import_seufisio.py --dry-run

# Importação (idempotente)
python3 scripts/import_seufisio.py

# Testes do script
cd scripts && python3 -m unittest test_import_seufisio -v
Segurança Tokens, dumps de pacientes e scripts/.env estão no .gitignore. Logs e --dry-run mascaram CPF/e-mail/nome para evitar vazamento de PII em stdout/CI.

15 Testes

Suítes organizadas por camada — unitários (Mockito), JPA (@DataJpaTest), controllers (@WebMvcTest) e integração (@SpringBootTest + H2 + MockMvc).

Unitários
Services e helpers com Mockito (PacienteService, PlanoService, PagamentoService, AulaService, etc.).
Controllers
@WebMvcTest + MockMvc cobrindo status, validação e serialização.
JPA
@DataJpaTest validando queries derivadas, índices e @Query consolidadas.
Integração
@SpringBootTest + H2 + MockMvc — segurança, scheduler, actuator e atomicidade de pagamento.

Execução

JAVA_HOME=/caminho/para/jdk21 mvn test

O @SpringBootTest usa H2 em memória automaticamente via src/test/resources/application.properties. O Flyway fica desabilitado nos testes.
